## Frequently occurring

<CardsGroup type="content">

<Card type="content">
##### ++101++<br/>
**Declined by Bank**<br/>
Generally insufficient funds or incorrect card data supplied, for example, expiry date or security code.
</Card>

<Card type="content">
##### ++501++<br/>
**This transaction has already been processed!**<br/>
Each transaction must have a unique Order ID. This includes those that have been processed but were declined (for example, a 101 Decline).
</Card>

<Card type="content">
##### ++505++ <br/>
**You cannot use this service from there [[URL/IP]].**<br/>
In Production transactions will only be accepted from designated IPs or URLs. If a transaction doesn't, it will be blocked.
</Card>

<Card type="content">
##### ++505++<br/>
**The refund password you entered was incorrect.**<br/>
Refunds (Rebates) and Credits must be signed with a unique SHA-1 hash. This error indicates that the hashed value was incorrect.
</Card>

<Card type="content">
##### ++508++<br/>
**Incorrect hash. Please check your code and the Developers Documentation.**<br/>
The hash included in the request was not the one expected. Usually indicates it has not has been constructed correctly or the Shared Secret is wrong. (505 for API)
</Card>

<Card type="content">
##### ++511++<br/>
**Unable to connect to the merchant's response URL [URL].**<br/>
The HPP was unable to connect to the Response URL. Your application or website will not have received the transaction response. Common causes of this include firewall rules or server availability.
</Card>

</CardsGroup>

| Solution | Error No. | Description |
| --- | --- | --- |
| API/HPP | 00  | Successful – The transaction has been authorized successfully; you may proceed with the sale. |
| API/HPP | 101 | Decline - Generally insufficient funds or incorrect card data supplied, for example, expiry date or security code. |
| API/HPP | 102 | Referral B - Contact the Authorization Center. Usually treated as a standard decline for Ecommerce systems, the transaction can still be processed using a Manual request once an authorization code is obtained. |
| API/HPP | 103 | Referral A - The card has been reported lost or stolen. Usually treated as a standard decline for ecommerce systems. |
| API/HPP | 111 | Soft Decline - Indicates that the Issuer requires Strong Customer Authentication (SCA) for a transaction. In order to avoid this outcome, please ensure you enable 3D Secure 2. |
| API/HPP | 120 | Return when the settlement amount is 200% or over of the original authorized amount. Only applicable to test accounts domiciled in North America (CA/US). |
| API/HPP | 121 | Return when the settlement amount is 25% or less of the original authorized amount. Only applicable to test accounts domiciled in North America (CA/US). |
| API/HPP | 13x | Response from the Payment Decline Reason Code (PDRC) service. See our [PDRC guide](/ecommerce/expand-your-integration/pdrc). |
| API/HPP | 2xx | Error with bank systems – generally you can tell the customer to try again later. The resolution time depends on the issue. |
| API/HPP | 205 | Comms Error - installment connection error. |
| API/HPP | 3xx | Error with Global Payments systems – generally you can tell the customer to try again later. The resolution time depends on the issue. |
| API/HPP | 5xx | Invalid content in request message. These can be due to a formatting issue or account configuration. Please see the various 5xx examples below. |
| API/HPP | 666 | Account deactivated – your Global Payments account has been suspended. Please contact Global Payments. |
